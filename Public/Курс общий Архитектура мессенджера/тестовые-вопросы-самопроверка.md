## Раздел 1 · Что такое мессенджер с точки зрения архитектуры
↩ [[Раздел 1. Что такое мессенджер с точки зрения архитектуры (расширенный)]]

**1.** Чем мессенджер архитектурно отличается от обычного веб-сервиса «запрос — ответ»?
- А) Ничем
- Б) Сообщение надо доставить тому, кто прямо сейчас ничего не спрашивает, поэтому сервер должен сам толкать данные на устройство — нужно постоянное соединение
- В) Мессенджер не использует серверы
- Г) В мессенджере нет базы данных

**2.** Что лежит в основе базового скелета мессенджера?
- А) Один сервер на всё
- Б) Только база данных
- В) Только клиент
- Г) Клиент → балансировщик → сервер соединений → Message-сервис → база и Реестр сессий → сервер соединений получателя → клиент

**3.** Какова роль Message-сервиса?
- А) Хранить медиа
- Б) Держать живые соединения
- В) Центральная точка логики: сохраняет сообщение, смотрит в Реестр сессий, где получатель, и пересылает на его сервер соединений
- Г) Шифровать переписку

**4.** Почему подсистемы (хранение, медиа, присутствие, fan-out, гео) выносят отдельно?
- А) Потому что у каждой свой характер нагрузки и свои компромиссы, масштабируются по-разному
- Б) Для красоты схемы
- В) Так требует закон
- Г) Чтобы было больше серверов

**5.** С чего НЕ стоит начинать проектирование?
- А) С уточнения требований
- Б) Сразу с рисования сложной схемы со всеми подсистемами
- В) С базового скелета и одного сообщения по нему
- Г) С прикидки нагрузки

**6.** Что значит «сервер сам толкает данные»?
- А) Клиент постоянно опрашивает сервер
- Б) Сервер удаляет сообщения
- В) Данные хранятся только на клиенте
- Г) Как только сообщение появилось, сервер по заранее открытому соединению отправляет его получателю, не дожидаясь запроса

> [!success]- Ответы и разбор
> 1. **Б** — инициатива на стороне сервера: он должен дотолкнуть данные тому, кто не спрашивает. Отсюда постоянное соединение.
> 2. **Г** — это и есть базовый путь сообщения, от которого наращивают подсистемы.
> 3. **В** — Message-сервис: сохранить, найти получателя в Реестре сессий, переслать на его узел.
> 4. **А** — у каждой подсистемы своя нагрузка и компромиссы; смешивать вредно.
> 5. **Б** — не начинать с рисования; сперва требования, рамки, скелет.
> 6. **Г** — push: сервер отправляет сразу по открытому соединению, без запроса клиента.

## Раздел 2 · Требования и рамки задачи
↩ [[Раздел 2. Требования и рамки задачи (расширенный)]]

**1.** С чего начать, когда просят «спроектируй мессенджер»?
- А) Сразу рисовать архитектуру
- Б) Писать код
- В) С уточняющих вопросов: scope, масштаб (DAU), гарантии по задержке, потерям и порядку
- Г) С выбора базы данных

**2.** Как прикинуть нагрузку в сообщениях в секунду?
- А) DAU × среднее число сообщений на пользователя → за день → делим на 86 400 → средний QPS → умножаем на пиковый коэффициент
- Б) Берём случайное число
- В) Считаем только хранилище
- Г) Спрашиваем у пользователя точное число

**3.** Что обычно доминирует в объёме хранилища?
- А) Текстовые сообщения
- Б) Статусы присутствия
- В) Логи
- Г) Медиа — поэтому ему нужно отдельное хранилище и CDN

**4.** Что выбирает мессенджер в дилемме «консистентность или доступность»?
- А) Строгую консистентность любой ценой
- Б) Склоняется к доступности: доставить сейчас, согласовать порядок/статусы чуть позже, но durability не ослабляет
- В) Полную остановку при разрыве
- Г) Удаление расходящихся данных

**5.** Что НЕ относится к нефункциональным требованиям?
- А) Задержка
- Б) Масштаб (нагрузка)
- В) Список поддерживаемых типов сообщений (это функциональное требование)
- Г) Гарантии по потерям

**6.** Зачем проектировать под пик, а не под средний QPS?
- А) Нагрузка неравномерна, система должна выдерживать пиковые моменты, иначе ляжет в час пик
- Б) Пик не важен
- В) Чтобы сэкономить серверы
- Г) Средний QPS всегда равен пику

> [!success]- Ответы и разбор
> 1. **В** — сначала сузить задачу уточняющими вопросами, а не рисовать.
> 2. **А** — DAU × сообщения → /86 400 → средний QPS → ×пиковый коэффициент. Важен порядок величины.
> 3. **Г** — медиа доминирует, ему нужны отдельное хранилище и CDN.
> 4. **Б** — склон к доступности, но сохранность (durability) свята.
> 5. **В** — типы сообщений это функциональное требование, остальное — нефункциональные.
> 6. **А** — система должна держать пик, иначе падает в час пик.

## Раздел 3 · Как клиент держит связь с сервером
↩ [[Раздел 3. Как клиент держит связь с сервером (расширенный)]]

**1.** Почему опрос (polling) не подходит мессенджеру?
- А) Он слишком надёжен
- Б) Он шифрует данные
- В) Он не работает в интернете
- Г) Частый опрос — лавина почти всегда пустых запросов (сеть, процессор, батарея), а редкий даёт задержку

**2.** Что используют вместо опроса?
- А) Электронную почту
- Б) Постоянное двустороннее соединение — WebSocket: одно рукопожатие, дальше полнодуплексный канал
- В) Один разовый запрос
- Г) SMS

**3.** Зачем отделять серверы соединений от Message-сервиса?
- А) У них разная природа: сервер соединений stateful (держит живые сокеты, тяжело масштабировать), а логику хотим держать stateless
- Б) Чтобы запутать схему
- В) Они одинаковые, делить незачем
- Г) Чтобы шифровать сообщения

**4.** Что такое Реестр сессий?
- А) База для медиа
- Б) Очередь сообщений
- В) Быстрое общее хранилище в памяти (типа Redis), где записано, на каком узле сейчас висит пользователь или устройство
- Г) Сервер для звонков

**5.** Как сервер понимает, куда переслать сообщение получателю?
- А) Угадывает
- Б) Спрашивает у отправителя
- В) Рассылает всем узлам
- Г) Смотрит в Реестр сессий, находит узел получателя и пересылает туда по внутренней сети

**6.** Что удобнее для рассылки многим (группы)?
- А) Искать каждого в реестре по отдельности
- Б) Шина публикаций-подписок: узлы подписаны на своих пользователей, отправитель публикует один раз
- В) Отправлять по электронной почте
- Г) Не отправлять вообще

> [!success]- Ответы и разбор
> 1. **Г** — polling расточителен (пустые запросы) либо даёт задержку. Нужна инициатива сервера.
> 2. **Б** — WebSocket: одно рукопожатие, дальше постоянный полнодуплексный канал.
> 3. **А** — stateful держатель соединений отделяют от stateless логики ради масштабирования.
> 4. **В** — Реестр сессий: быстрый справочник «пользователь/устройство → узел».
> 5. **Г** — через Реестр сессий находим узел и пересылаем по внутренней сети.
> 6. **Б** — для многих удобнее шина публикаций-подписок, а не поиск каждого.

## Раздел 4 · Путь сообщения: онлайн-доставка
↩ [[Раздел 4. Путь сообщения онлайн-доставка]]

**1.** Что делает Message-сервис, получив сообщение от Алисы для Боба?
- А) Сразу удаляет его
- Б) Отправляет его всем пользователям
- В) Присваивает номер, сохраняет в базу, смотрит в Реестре сессий, где Боб, и пересылает на его сервер соединений
- Г) Ждёт, пока Боб сам опросит сервер

**2.** Где в пути появляется галочка «отправлено»?
- А) Когда Боб прочитал сообщение
- Б) Когда устройство Боба получило сообщение
- В) Когда сервер принял и надёжно сохранил сообщение
- Г) Никогда

**3.** Что означает галочка «доставлено»?
- А) Сообщение удалено
- Б) Сервер сохранил сообщение
- В) Алиса открыла чат
- Г) Устройство Боба получило сообщение и прислало обратно квитанцию

**4.** Что означает галочка «прочитано»?
- А) Боб открыл чат, и его устройство прислало вторую квитанцию
- Б) Сервер принял сообщение
- В) Сообщение зашифровано
- Г) Сообщение в очереди

**5.** Зачем сообщение сохраняют в базу ещё до доставки?
- А) Чтобы оно не потерялось, если получатель офлайн или доставка сорвётся
- Б) Чтобы занять место
- В) Чтобы Боб не смог его прочитать
- Г) Низачем

**6.** Что произойдёт, если Боб отключился прямо в момент отправки?
- А) Сообщение теряется навсегда
- Б) Падает вся система
- В) Сообщение уходит в очередь ожидания, а Боба будит уведомление — это просто переход на офлайн-ветку
- Г) Алиса должна отправить заново вручную

> [!success]- Ответы и разбор
> 1. **В** — Message-сервис: номер → сохранить → Реестр сессий → переслать на узел Боба.
> 2. **В** — «отправлено» = сервер принял и сохранил сообщение.
> 3. **Г** — «доставлено» = устройство получателя получило и прислало квитанцию.
> 4. **А** — «прочитано» = Боб открыл чат, пришла вторая квитанция.
> 5. **А** — сохранение до доставки гарантирует, что сообщение не потеряется.
> 6. **В** — это не ошибка, а переход на офлайн-доставку: очередь + пуш.

## Раздел 5 · Офлайн-доставка и пуши
↩ [[Раздел 5. Офлайн-доставка и пуши]]

**1.** Куда девается сообщение, если получатель офлайн?
- А) Удаляется
- Б) Сохраняется на сервере и кладётся в очередь ожидания (личный ящик получателя), хранится с сохранением порядка
- В) Отправляется обратно отправителю
- Г) Теряется

**2.** Как получатель получит сообщения при возвращении в сеть?
- А) Просит собеседника переслать заново
- Б) Никак
- В) Только через неделю
- Г) Устройство переподключается, сервер отдаёт накопленное из очереди по порядку, устройство подтверждает квитанциями

**3.** Зачем нужны APNs/FCM, если у нас есть своё соединение?
- А) Своё соединение живёт, только пока приложение открыто; разбудить закрытое приложение может лишь системная служба уведомлений (APNs у Apple, FCM у Google)
- Б) Чтобы шифровать
- В) Чтобы хранить историю
- Г) Они не нужны

**4.** Что именно делает пуш-уведомление через APNs/FCM?
- А) Доставляет само сообщение целиком
- Б) Удаляет очередь
- В) Будит приложение, чтобы оно переподключилось и забрало сообщение из очереди
- Г) Шифрует переписку

**5.** Почему при сквозном шифровании в пуш не кладут текст сообщения?
- А) Потому что текст слишком длинный
- Б) Потому что сервер не видит содержимого и не может положить его в пуш
- В) Потому что пуши запрещены
- Г) Потому что так быстрее

**6.** Что хранит очередь ожидания?
- А) Только медиа
- Б) Пароли
- В) Только онлайн-статусы
- Г) Накопленные для офлайн-получателя сообщения с сохранением порядка, пока он не заберёт их

> [!success]- Ответы и разбор
> 1. **Б** — сообщение уже сохранено; кладётся в очередь (ящик) получателя по порядку.
> 2. **Г** — при возвращении устройство забирает накопленное из очереди, подтверждая квитанциями.
> 3. **А** — закрытое приложение будит только системная служба (APNs/FCM).
> 4. **В** — пуш будит приложение, а не доставляет сообщение сам.
> 5. **Б** — сервер не видит содержимого при E2EE, поэтому текст в пуш не кладёт.
> 6. **Г** — очередь хранит сообщения офлайн-получателя по порядку до получения.

## Раздел 6 · Гарантии доставки и порядок
↩ [[Раздел 6. Гарантии доставки и порядок]]

**1.** Как гарантировать, что сообщение не потеряется?
- А) Надеяться на сеть
- Б) Сначала надёжно сохранить, затем доставлять по принципу «хотя бы один раз», повторяя до подтверждения
- В) Отправить один раз и забыть
- Г) Не сохранять вообще

**2.** Откуда берутся дубликаты и как с ними бороться?
- А) Дубликатов не бывает
- Б) Их создаёт получатель
- В) Повторы при «хотя бы один раз» дают дубликаты; борются дедупликацией по уникальному номеру сообщения (идемпотентность)
- Г) Удалением всей переписки

**3.** Как обеспечить правильный порядок сообщений?
- А) По времени на сервере
- Б) По алфавиту
- В) Случайно
- Г) Возрастающим порядковым номером в рамках конкретного чата, по которому устройство сортирует

**4.** Почему нельзя полагаться на время сервера для порядка?
- А) Часы серверов расходятся, поэтому время ненадёжно для упорядочивания
- Б) Время всегда точное
- В) Время шифруется
- Г) У серверов нет часов

**5.** Что выбирают на практике — at-least-once или честный exactly-once?
- А) Честный exactly-once любой ценой
- Б) Ничего из этого
- В) At-least-once + дедупликация по номерам, что даёт эффект exactly-once
- Г) At-most-once с потерями

**6.** Где нужен порядок — глобально или внутри чата?
- А) Глобально по всему мессенджеру
- Б) Внутри чата — поэтому единый мировой счётчик не нужен, достаточно номеров в рамках чата
- В) Порядок вообще не нужен
- Г) Только для медиа

> [!success]- Ответы и разбор
> 1. **Б** — durability (сохранить) + at-least-once (повторять до подтверждения).
> 2. **В** — повторы дают дубликаты; лечит дедупликация по уникальному message_id.
> 3. **Г** — порядковый номер в рамках чата, по нему сортирует устройство.
> 4. **А** — часы серверов расходятся, время для порядка ненадёжно.
> 5. **В** — at-least-once + дедуп = эффект exactly-once; честный exactly-once на ненадёжной сети недостижим.
> 6. **Б** — порядок нужен внутри чата, глобальный счётчик не требуется.

## Раздел 7 · Статусы присутствия и индикаторы
↩ [[Раздел 7. Статусы присутствия и индикаторы]]

**1.** Как сделать онлайн-статус для миллионов?
- А) Хранить его в надёжной базе на каждое изменение
- Б) Спрашивать у пользователя
- В) Выводить из наличия живой связи на сервере соединений, обрыв ловить по пропаже сигналов «я жив»
- Г) Не делать вообще

**2.** Как устроено «печатает…»?
- А) Пишется в базу на каждое нажатие клавиши
- Б) Чисто эфемерно: ничего не пишем в базу, агрегируем события (раз в пару секунд) и гасим по таймауту
- В) Хранится вечно
- Г) Рассылается всем пользователям мессенджера

**3.** Почему присутствие — серьёзная нагрузка?
- А) Это редкое событие
- Б) Это постоянный поток мелких обновлений, который надо размножать на всех наблюдателей; умножь на миллионы — лавина
- В) Оно не нагружает систему
- Г) Потому что шифруется

**4.** Кому рассылать статусы присутствия?
- А) Только тем, кто сейчас реально смотрит на пользователя (открыл чат), а не всем подряд
- Б) Всем пользователям мессенджера
- В) Никому
- Г) Только администраторам

**5.** Что будет, если статус присутствия слегка «врёт»?
- А) Падает вся система
- Б) Теряются сообщения
- В) Ничего критичного — присутствие приблизительно, доставку сообщений это не ломает
- Г) Пользователя отключают

**6.** Где лучше хранить данные присутствия?
- А) В надёжной базе с репликацией
- Б) В CDN
- В) В объектном хранилище
- Г) В быстрой памяти, а не в надёжной базе — это дёшево и допускает небольшую неточность

> [!success]- Ответы и разбор
> 1. **В** — онлайн-статус выводят из наличия живого соединения, обрыв — по пропаже heartbeat.
> 2. **Б** — «печатает…» эфемерно: агрегация + таймаут, без записи в базу.
> 3. **Б** — постоянный поток мелких обновлений × миллионы наблюдателей = лавина.
> 4. **А** — только активным наблюдателям, а не всем.
> 5. **В** — присутствие приблизительно; неточность не ломает доставку.
> 6. **Г** — быстрая память, не надёжная база; допустима неточность.

## Раздел 8 · Где живут сообщения и как шардируются
↩ [[Раздел 8. Где живут сообщения и как шардируются]]

**1.** Где и как хранятся сообщения?
- А) В оперативной памяти клиента
- Б) В отдельном хранилище под нагрузку «много записей, простое чтение по чату, бесконечный рост», шардированном и реплицированном
- В) В одном файле на одном сервере
- Г) В CDN

**2.** По какому ключу обычно шардируют историю?
- А) По цвету интерфейса
- Б) По имени пользователя
- В) По идентификатору чата — тогда все сообщения чата лежат на одном шарде, и чтение последних бьёт в один сервер
- Г) Случайно

**3.** Как гарантируется durability хранения?
- А) Репликацией: сообщение пишется на несколько копий, и запись успешна только после подтверждения от нескольких из них
- Б) Одной копией
- В) Удалением старого
- Г) Кэшированием

**4.** SQL или NoSQL для истории сообщений — и почему?
- А) Только SQL всегда
- Б) Только клиентская база
- В) Для истории — NoSQL-стиль (огромный поток записей и рост хорошо ложатся на горизонтальное масштабирование), но аккаунты/контакты могут жить в SQL
- Г) Никакой базы не нужно

**5.** Что такое «горячий шард» и где он возникает?
- А) Перегретый сервер из-за жары
- Б) Шард, в который бьёт непропорционально много нагрузки — например, чат-гигант (большой канал); решают особой обработкой
- В) Шард с медиа
- Г) Резервная копия

**6.** Почему база сообщений оптимизирована под добавление?
- А) Потому что сообщения часто редактируют
- Б) Потому что сообщения в основном дописываются и почти не меняются — нагрузка «много записей в конец»
- В) Потому что так требует закон
- Г) Потому что это SQL

> [!success]- Ответы и разбор
> 1. **Б** — отдельное хранилище под профиль нагрузки мессенджера, шардированное и реплицированное.
> 2. **В** — шард по chat_id: сообщения чата вместе, чтение последних — в один шард.
> 3. **А** — durability через репликацию и подтверждение от нескольких копий.
> 4. **В** — история в NoSQL-стиле; аккаунты/связи могут быть в SQL. Инструмент под тип данных.
> 5. **Б** — горячий шард = чат-гигант; лечится особой обработкой больших каналов.
> 6. **Б** — сообщения дописываются и почти не редактируются, отсюда оптимизация под добавление.

## Раздел 9 · Медиа и CDN
↩ [[Раздел 9. Медиа и CDN]]

**1.** Почему медиа нельзя гонять тем же путём, что и текст?
- А) Потому что медиа нельзя шифровать
- Б) Потому что текст важнее медиа
- В) Потому что файлы большие (мегабайты и гигабайты), а канал сообщений и база заточены под крошечный текст — большой файл их забьёт
- Г) Никакой разницы нет

**2.** Как устроена отправка большого видео?
- А) Видео целиком кладётся внутрь текстового сообщения
- Б) Файл загружается отдельно в объектное хранилище, а в сообщении пересылается лишь ссылка (плюс размер, тип, превью)
- В) Видео разбивается на буквы и шлётся по одной
- Г) Видео хранится только на устройстве отправителя

**3.** Что такое объектное хранилище?
- А) Обычная SQL-база для сообщений
- Б) Оперативная память сервера
- В) Папка на телефоне
- Г) Специальное хранилище для больших файлов, где каждый файл — «объект» с адресом; дёшево и почти бесконечно масштабируется

**4.** Почему в сообщении хранят ссылку, а не сам файл?
- А) Сообщение остаётся крошечным (быстрая доставка, дешёвое хранение), а файл лежит один раз и ссылку можно переслать многим
- Б) Потому что ссылки красивее
- В) Потому что файлы запрещено хранить на сервере
- Г) Чтобы сообщение нельзя было прочитать

**5.** Зачем нужен CDN?
- А) Чтобы шифровать файлы
- Б) Чтобы удалять старые сообщения
- В) Это сеть серверов по всему миру с копиями файлов: пользователь качает с ближайшего сервера — быстро, и с основного хранилища снимается нагрузка
- Г) Чтобы хранить пароли

**6.** Что такое транскодинг?
- А) Шифрование видео перед отправкой
- Б) Перекодирование медиа в разные форматы и качества под устройства и скорость сети
- В) Сжатие текстовых сообщений
- Г) Подсчёт числа просмотров

> [!success]- Ответы и разбор
> 1. **В** — файлы огромны, путь сообщений и база под них не годятся.
> 2. **Б** — файл грузится отдельно, в сообщении лишь ссылка с превью и описанием.
> 3. **Г** — объектное хранилище: дешёвое масштабируемое место для больших файлов-объектов.
> 4. **А** — ссылка держит сообщение крошечным, файл хранится один раз.
> 5. **В** — CDN: копии поближе к пользователю + разгрузка хранилища.
> 6. **Б** — транскодинг = перекодирование в разные качества/форматы, делается в фоне.

- В) Устройства не связаны

- Б) Ноутбук сам угадает
- В) Только если переслать сообщение заново

- Б) На нескольких устройствах нельзя шифровать

- Г) Мультидевайс никак не влияет

> 2. **А** — источник правды на сервере, устройства — копии.
> 3. **В** — устройства ненадёжны как хранилище общей истории.

> 5. **Г** — прочитанность это событие; синхронизируется через сервер.
> 6. **А** — у каждого устройства свои ключи, шифровать надо под каждое.

## Раздел 11 · Группы и фан-аут
↩ [[Раздел 11. Группы и фан-аут]]

**1.** Что такое fan-out в группах?
- А) Шифрование группового чата
- Б) Размножение одного сообщения на всех получателей группы или канала
- В) Удаление участника
- Г) Сжатие сообщений

**2.** Что такое «fan-out on write» (размножать при отправке)?
- А) Хранить сообщение один раз, получатели читают из общего места
- Б) Не доставлять вообще
- В) В момент отправки сразу класть копию сообщения в ящик каждого получателя
- Г) Шифровать при записи

**3.** В чём минус «fan-out on write» для огромного канала?
- А) Одно сообщение порождает столько записей, сколько подписчиков — для 10 млн это 10 млн записей за пост
- Б) Чтение становится очень медленным
- В) Сообщение нельзя зашифровать
- Г) Минусов нет

**4.** Что такое «fan-out on read» (собирать при чтении)?
- А) Копировать сообщение каждому при отправке
- Б) Доставлять только онлайн-участникам
- В) Удалять сообщение после прочтения
- Г) Хранить сообщение один раз в общем месте, участники сами читают его при открытии чата

**5.** Как разумно сочетать оба подхода?
- А) Всегда только fan-out on write
- Б) Малые/средние группы — fan-out on write (дёшево размножить, мгновенное чтение); огромные каналы — fan-out on read (хранить один раз, подтягивать)
- В) Всегда только fan-out on read
- Г) Выбор случайный

**6.** Почему рассылка в канал на миллион — отдельная задача?
- А) Миллион сообщений нельзя зашифровать
- Б) Каналы запрещены
- В) Размножать пост на миллион при каждой отправке неподъёмно; пост хранят один раз и доставляют «лениво», пользуясь тем, что большинство неактивны
- Г) В канале нет сообщений

> [!success]- Ответы и разбор
> 1. **Б** — fan-out = размножение сообщения на всех получателей.
> 2. **В** — fan-out on write: копия в ящик каждого в момент отправки.
> 3. **А** — на миллионник один пост = миллионы записей; дороговизна записи убивает подход.
> 4. **Г** — fan-out on read: хранить один раз, читатели подтягивают.
> 5. **Б** — гибрид по размеру: малые — on write, гиганты — on read.
> 6. **В** — пост хранят один раз, доставляют лениво, опираясь на неактивность большинства.

## Раздел 12 · Балансировка, шлюзы и маршрутизация соединений
↩ [[Раздел 12. Балансировка, шлюзы и маршрутизация соединений]]

**1.** Как сервер A отправит сообщение получателю, висящему на сервере Б?
- А) Посмотрит в Реестре сессий узел получателя и перешлёт на узел Б по внутренней сети — тот протолкнёт клиенту
- Б) Узел A напрямую знает все соединения всех узлов
- В) Разошлёт всем узлам сразу
- Г) Дождётся, пока получатель сам опросит сервер

**2.** Как балансируют постоянные соединения?
- А) Перекидывают каждый кадр на новый случайный сервер
- Б) Не балансируют
- В) Складывают все соединения на один сервер
- Г) Распределяют по узлам (обычно на транспортном уровне), каждое соединение прилипает к узлу на всё время жизни; цель — ровный разброс без горячих узлов

**3.** Зачем для групп шина публикаций-подписок (pub/sub), а не только реестр?
- А) Чтобы шифровать
- Б) Чтобы не искать каждого участника по отдельности: узлы подписываются на «каналы», издатель публикует один раз, нужные узлы получают сами
- В) Чтобы хранить историю
- Г) Шина не нужна

**4.** Что такое корректная деградация под перегрузкой?
- А) Система мгновенно полностью отключается
- Б) Система работает как ни в чём не бывало
- В) Сначала жертвует менее важным (статусы, «печатает…»), сохраняя главное — доставку сообщений
- Г) Все сообщения удаляются

**5.** Что происходит при падении connection-сервера?
- А) Все его соединения рвутся, клиенты переподключаются к другим узлам; система работает для остальных, пропущенное догрузят из очереди
- Б) Все сообщения теряются навсегда
- В) Падает вся система
- Г) Пользователи переезжают без разрыва соединения

**6.** Что делает клиент при переподключении?
- А) Ничего, просто ждёт
- Б) Начинает переписку с нуля
- В) Создаёт новый аккаунт
- Г) Заново регистрируется в реестре, переподписывается и догружает пропущенное из очереди по номеру «докуда получено»

> [!success]- Ответы и разбор
> 1. **А** — маршрут через Реестр сессий, пересылка на узел получателя.
> 2. **Г** — соединения на транспортном уровне, прилипают к узлу; цель — ровный разброс.
> 3. **Б** — шина избавляет от поиска каждого: подписка + одна публикация.
> 4. **В** — деградация: жертвуем второстепенным, сохраняя доставку.
> 5. **А** — соединения рвутся, клиенты идут на другие узлы; данные в очереди.
> 6. **Г** — перерегистрация, переподписка, догрузка по номеру; идемпотентность от дублей.

## Раздел 13 · Геораспределение и доступность

**1.** Зачем несколько дата-центров в разных регионах?

- В) Снизить задержку (подключение к ближайшему) и пережить отказ целого дата-центра

- В) В другой стране хуже интернет

- Б) Только медиа
- В) Только пароли
- Г) Копии данных (история, очереди) — чтобы читать локально и пережить потерю дата-центра

- Г) Репликация удваивает скорость интернета

**5.** При разрыве связи между регионами мессенджер чаще выбирает…
- А) Полностью остановиться
- Б) Удалить расходящиеся данные
- В) Доступность: отвечать из локальной (возможно, чуть устаревшей) копии, согласовать позже; сохранность сообщения не ослабляют
- Г) Строгую консистентность любой ценой

**6.** Как система ведёт себя при потере целого дата-центра?

- Б) Все данные пропадают
- В) Система умирает до ремонта
- Г) Пользователи работают, не переподключаясь

> [!success]- Ответы и разбор

> 3. **Г** — копии данных ради локального чтения и выживания при потере дата-центра.
> 4. **Б** — копирование через океан небыстрое, удалённая копия отстаёт.
> 5. **В** — склон к доступности, но durability сообщения сохраняют.
> 6. **А** — переключение на регион с копиями, переподключение; данные уцелели.

## Раздел 14 · Транспортное против сквозного шифрования
↩ [[Раздел 14. Транспортное против сквозного шифрования]]

**1.** Что шифрует транспортное шифрование (TLS)?
- А) Само сообщение от устройства до устройства
- Б) Только пароли
- В) Ничего, это просто название
- Г) Канал связи между устройством и сервером — в пути защищено, но на сервере расшифровывается

**2.** Что такое сквозное шифрование (E2EE)?
- А) Шифрование только на сервере
- Б) Сообщение шифруется на устройстве отправителя и расшифровывается только на устройстве получателя; по пути, включая сервер, его никто не читает
- В) Шифрование, которое сервер может снять
- Г) То же, что транспортное

**3.** Что видит сервер при ТОЛЬКО транспортном шифровании?
- А) Полное содержимое сообщений — расшифровывает у себя, хранит и может читать
- Б) Только зашифрованную кашу
- В) Ничего
- Г) Только размер

**4.** Что видит сервер при сквозном шифровании?
- А) Полный текст
- Б) Текст с разрешения пользователя
- В) Только зашифрованные байты (шифртекст) — содержимое расшифровать не может, хотя хранит и доставляет
- Г) Сервер не участвует в доставке

**5.** От какой угрозы защищает именно сквозное шифрование (а транспортное — нет)?
- А) От разряда батареи
- Б) От медленного интернета
- В) От опечаток
- Г) От самого оператора/сервера и его взлома: оператор не может прочитать или выдать переписку, не видя её

**6.** Что сквозное шифрование НЕ защищает?
- А) Содержимое в пути
- Б) Не спасает при взломе самого устройства или сливе собеседником; не скрывает метаданные (кто-кому-когда)
- В) Текст от чтения сервером
- Г) Оно защищает от всего

> [!success]- Ответы и разбор
> 1. **Г** — TLS шифрует канал «устройство — сервер»; на сервере расшифровывается.
> 2. **Б** — E2EE: от устройства до устройства, посередине никто не читает.
> 3. **А** — при только транспортном сервер видит полное содержимое.
> 4. **В** — при сквозном сервер видит лишь шифртекст.
> 5. **Г** — E2EE добавляет защиту от оператора и взлома сервера.
> 6. **Б** — не закрывает конечные точки и метаданные.

## Раздел 15 · Signal Protocol, X3DH и установка ключей
↩ [[Раздел 15. Signal Protocol, X3DH и установка ключей из названия]]

**1.** Какую задачу решает установка ключей X3DH?
- А) Сжать сообщение
- Б) Как двум устройствам, которые не встречались и не могут доверить ключ серверу, договориться об общем секретном ключе — да ещё когда они не онлайн одновременно
- В) Ускорить доставку
- Г) Хранить историю

**2.** В чём суть обмена ключами Диффи — Хеллмана (без математики)?
- А) Стороны обмениваются только публичными частями, но, скомбинировав свою секретную с чужой публичной, обе получают один и тот же общий секрет, который подслушивающий вычислить не может
- Б) Пересылают секретный ключ через сервер
- В) Сервер генерирует ключ и раздаёт обоим
- Г) Ключ диктуют по телефону

**3.** Зачем сервер хранит prekeys (заранее загруженные публичные ключи)?
- А) Чтобы читать сообщения
- Б) Чтобы экономить место
- В) Чтобы Алиса установила общий ключ и отправила шифрованное сообщение, даже когда Боб офлайн: она берёт его prekeys с сервера
- Г) Чтобы ускорить интернет

**4.** Что сервер НЕ знает при установке ключей по X3DH?
- А) Публичные ключи пользователей
- Б) Факт, что Алиса написала Бобу
- В) Размер сообщения
- Г) Секретные (приватные) части ключей и сам общий секрет — они не покидают устройства

**5.** Почему обычного «живого» обмена ключами недостаточно?
- А) Он слишком быстрый
- Б) Получатель часто офлайн, а живой обмен требует обоих онлайн; prekeys позволяют «договориться» с тем, кого нет в сети
- В) Сервер запрещает обмен
- Г) Живой обмен нельзя зашифровать

**6.** Какому риску всё же приходится доверять серверу?
- А) Что сервер выдаст правильные публичные ключи, а не ключи самозванца («человек посередине»); поэтому дают сверить «код безопасности» вне мессенджера
- Б) Что не разрядит батарею
- В) Что быстро доставит сообщение
- Г) Никакому

> [!success]- Ответы и разбор
> 1. **Б** — тройная сложность: незнакомцы, ключ нельзя отдать серверу, не онлайн одновременно.
> 2. **А** — обмен только публичными частями, секрет рождается у обоих, по сети не передаётся.
> 3. **В** — prekeys делают рукопожатие асинхронным (Боб офлайн).
> 4. **Г** — приватные части и общий секрет не покидают устройств.
> 5. **Б** — живой обмен требует обоих онлайн; prekeys решают это.
> 6. **А** — доверие к раздаче правильных публичных ключей; закрывают сверкой кода безопасности.

## Раздел 16 · Double Ratchet и forward secrecy
↩ [[Раздел 16. Double Ratchet и forward secrecy]]

**1.** Что делает Double Ratchet с ключом?
- А) Использует один ключ навсегда
- Б) Шифрует только первое сообщение
- В) Хранит все ключи вечно
- Г) Меняет ключ на каждое сообщение, старый сразу удаляет; «храповик» крутится только вперёд

**2.** Что такое forward secrecy?
- А) Сообщения доставляются быстрее
- Б) Сервер видит переписку
- В) Если злоумышленник украл ключ сегодня, он НЕ сможет прочитать прошлые сообщения — их ключи уже удалены
- Г) Ключ один на всех

**3.** Почему утечка одного ключа не раскрывает всю переписку?
- А) Переписка не шифруется
- Б) У каждого сообщения свой одноразовый ключ, удаляемый после использования, — украденный ключ к чужим сообщениям не подходит
- В) Сервер хранит резервную копию ключей
- Г) Ключ нельзя украсть

**4.** Что такое post-compromise security?
- А) Даже если ключ временно добыли, после следующего обмена свежим материалом защита самовосстанавливается, и будущие сообщения снова недоступны
- Б) После взлома уже ничего не спасти
- В) Ключи никогда не меняются
- Г) Сервер восстанавливает переписку

**5.** Forward secrecy и post-compromise security защищают…
- А) Оба только будущее
- Б) Оба только прошлое
- В) Ничего конкретного
- Г) Forward secrecy — прошлые сообщения, post-compromise security — восстанавливает защиту будущих

**6.** Зачем «двойной» храповик, а не один?
- А) Для красоты названия
- Б) Чтобы шифровать дважды один текст
- В) Один меняет ключ на каждое сообщение, второй подмешивает свежую случайность при обмене ответами — вместе дают защиту прошлого и восстановление будущего
- Г) Чтобы замедлить переписку

> [!success]- Ответы и разбор
> 1. **Г** — ключ прокручивается на каждое сообщение, старый удаляется, назад хода нет.
> 2. **В** — forward secrecy: кража ключа не раскрывает прошлое.
> 3. **Б** — у каждого сообщения свой одноразовый удаляемый ключ.
> 4. **А** — post-compromise security: защита самовосстанавливается после свежего обмена.
> 5. **Г** — forward secrecy бережёт прошлое, post-compromise — лечит будущее.
> 6. **В** — два механизма: ключ на сообщение + свежая случайность при диалоге.

## Раздел 17 · MTProto и подход Telegram
↩ [[Раздел 17. MTProto и подход Telegram]]

**1.** Чем подход Telegram отличается от Signal/WhatsApp по умолчанию?
- А) Два вида чатов: облачные (по умолчанию, НЕ сквозные) и секретные (по запросу, сквозные); у Signal/WhatsApp сквозное для всех по умолчанию
- Б) Telegram ничего не шифрует
- В) Telegram шифрует всё сквозным по умолчанию, как Signal
- Г) Разницы нет

**2.** Что такое облачные чаты в Telegram?
- А) Сквозно зашифрованы и не хранятся на сервере
- Б) Только для каналов
- В) Чаты по умолчанию: канал «клиент — сервер» шифруется, но на сервере они расшифровываются и хранятся — сервер может их прочитать
- Г) Доступны только офлайн

**3.** Что такое MTProto?
- А) Название мессенджера
- Б) Алгоритм сжатия видео
- В) Стандартный протокол, как у всех
- Г) Собственный (самописный) протокол безопасности Telegram (Mobile Transport Protocol); за нестандартность его критиковали криптографы

**4.** Зачем Telegram сделал облачные чаты не сквозными — какая выгода?
- А) Чтобы экономить электричество
- Б) Бесшовный мультидевайс, мгновенная синхронизация истории, доступ откуда угодно без бэкапов, серверный поиск по всем сообщениям
- В) Чтобы замедлить доставку
- Г) Никакой, это ошибка

**5.** Что такое секретные чаты в Telegram?
- А) Сквозные по запросу: только 1-на-1, привязаны к устройствам, не синхронизируются в облако, есть самоуничтожение
- Б) Обычные чаты по умолчанию
- В) Чаты, которые читает сервер
- Г) Групповые каналы

**6.** Как продакт объяснит выбор Telegram?
- А) Telegram не умеет сквозное шифрование
- Б) Случайное решение
- В) Осознанная ставка на удобство (облако, синхронизация, поиск) с E2EE как опцией; Signal/WhatsApp сделали обратную ставку. Это позиционирование между удобством и приватностью
- Г) Удобство и приватность — одно и то же

> [!success]- Ответы и разбор
> 1. **А** — у Telegram два типа чатов, сквозное только в секретных.
> 2. **В** — облачные по умолчанию: сервер может их читать.
> 3. **Г** — MTProto — самописный протокол, его критиковали.
> 4. **Б** — облако даёт мультидевайс, синхронизацию, поиск.
> 5. **А** — секретные: сквозные, по запросу, 1-на-1, привязаны к устройствам.
> 6. **В** — осознанное позиционирование между удобством и приватностью.

## Раздел 18 · Групповое E2EE
↩ [[Раздел 18. Групповое E2EE]]

**1.** Почему сквозное шифрование в группе сложнее, чем в диалоге?
- А) В группе сообщения длиннее
- Б) Группы не шифруются
- В) Шифровать надо для каждого получателя, а в группе их много — наивно это N отдельных шифрований на каждое сообщение
- Г) В группе нет ключей

**2.** Что такое sender keys?
- А) Один общий ключ на всю группу навсегда
- Б) Ключи, которые хранит сервер
- В) Пароль от группы
- Г) У каждого отправителя свой групповой ключ: раздаёт один раз всем, потом шифрует каждое сообщение однократно — все могут расшифровать

**3.** Зачем sender keys вместо «шифровать каждому отдельно каждый раз»?
- А) Чтобы платить за попарную раздачу один раз, а дальше шифровать каждое сообщение однократно — иначе на 1000 человек было бы 1000 шифрований на сообщение
- Б) Чтобы сервер мог читать
- В) Чтобы отключить шифрование
- Г) Чтобы замедлить группу

**4.** Что происходит с ключами, когда участник ВЫХОДИТ?
- А) Ничего, ключ сам исчезает
- Б) Оставшиеся перевыдают ключи отправителя, потому что у вышедшего остаётся старый ключ — иначе он читал бы будущие сообщения
- В) Группа удаляется
- Г) Сервер блокирует доступ, ключи не трогают

**5.** Почему удаление участника — дорогая операция в больших группах?
- А) Приходится пересылать всю историю
- Б) Сервер падает
- В) После каждого выхода надо перевыдать ключи всем оставшимся, а при частой смене состава это много операций
- Г) Удаление бесплатно

**6.** Что такое MLS и зачем он?
- А) Алгоритм сжатия
- Б) Название мессенджера
- В) То же, что sender keys
- Г) Современный стандартный протокол группового E2EE; раскладывает участников в «дерево», и перевыдача ключей дешевеет с N до примерно log N — масштабируется на большие группы

> [!success]- Ответы и разбор
> 1. **В** — шифрование под каждого получателя, а их много.
> 2. **Г** — sender keys: раздал один раз, дальше шифруешь однократно.
> 3. **А** — переносит дорогую попарную работу в разовую раздачу.
> 4. **Б** — у вышедшего остаётся копия ключей, поэтому перевыдают (сменить замок).
> 5. **В** — каждый выход запускает перевыдачу всем, при churn это лавина.
> 6. **Г** — MLS: дерево, перевыдача ≈ log N вместо N.

## Раздел 19 · Чего E2EE НЕ закрывает: метаданные и бэкапы
↩ [[Раздел 19. Чего E2EE НЕ закрывает, метаданные и бэкапы]]

**1.** Что такое метаданные в переписке?
- А) Сам текст сообщений
- Б) Только пароли
- В) Зашифрованное содержимое
- Г) Данные ВОКРУГ сообщения: кто кому писал, когда, как часто, сколько, откуда — но не само содержание

**2.** Что сервер видит при включённом сквозном шифровании?
- А) Полный текст
- Б) Содержимое не видит, но видит метаданные: кто кому, когда и как часто
- В) Вообще ничего
- Г) Только язык сообщения

**3.** Почему сервер неизбежно видит метаданные даже при E2EE?
- А) Шифрование слабое
- Б) Пользователь сам отправляет их открыто
- В) Серверу нужно маршрутизировать и доставлять сообщения — для этого он обязан знать отправителя, получателя и время
- Г) Он их не видит, это миф

**4.** Почему облачный бэкап может «сломать» E2EE?
- А) Бэкап — копия переписки вне защищённого канала; если он не зашифрован сквозным образом, содержимое лежит читаемым в облаке
- Б) Бэкап ускоряет расшифровку на сервере
- В) Бэкап стирает ключи
- Г) Бэкап не связан с приватностью

**5.** Где на самом деле самое слабое звено?
- А) В алгоритме шифрования
- Б) В скорости интернета
- В) В размере сообщения
- Г) В самих устройствах: на устройстве сообщения расшифрованы, поэтому шпионская программа, разблокированный телефон или скриншот собеседника обходят любое шифрование

**6.** Может ли твоя приватность зависеть от собеседника?
- А) Нет, только от меня
- Б) Да: твои сообщения лежат и на устройстве собеседника, и если ОН включит незашифрованный облачный бэкап, переписка окажется читаемой в его облаке
- В) Нет, его бэкап на меня не влияет
- Г) Да, но только если у него старый телефон

> [!success]- Ответы и разбор
> 1. **Г** — метаданные = «конверт»: кто, кому, когда, откуда.
> 2. **Б** — содержимое скрыто, метаданные видны.
> 3. **В** — без отправителя/получателя/времени доставить нельзя.
> 4. **А** — бэкап это копия вне канала; незашифрованный — дыра.
> 5. **Г** — конечные устройства: на них всё расшифровано.
> 6. **Б** — да, зависит и от настроек бэкапа собеседника.

## Раздел 20 · Регуляторный и продуктовый угол безопасности
↩ [[Раздел 20. Регуляторный и продуктовый угол безопасности]]

**1.** Какие три типичных требования предъявляют регуляторы?
- А) Доступ к переписке/ключам, локализация данных (хранение в стране), встраивание бэкдора
- Б) Ускорение интернета, снижение цены, новые стикеры
- В) Только перевод интерфейса
- Г) Запрет групповых чатов

**2.** Почему «бэкдор только для властей» — технически проблемная идея?
- А) Власти не умеют им пользоваться
- Б) Он замедляет приложение
- В) Бэкдор шифрует дважды
- Г) Лазейка, которая существует, может быть найдена и использована кем угодно — нельзя сделать дверь, открывающуюся только для «хороших»

**3.** Что честно может ответить компания с настоящим E2EE на запрос «дайте ключи»?
- А) «Сейчас расшифруем и пришлём всю переписку»
- Б) «У нас этих ключей нет — они только на устройствах пользователей; выполнить требование = сломать архитектуру для всех»
- В) «Ключи хранятся у нас в открытом виде, держите»
- Г) «Перешлите запрос в Apple»

**4.** В чём состоит позиционирование приватности как продукта (Signal)?
- А) В количестве стикеров
- Б) В скорости доставки
- В) Приватность — главная ценность бренда: «мы не можем прочитать вашу переписку»; держится на доверии — открытом протоколе, аудитах, прозрачности
- Г) В цене подписки

**5.** Почему доверие — ключевой актив для приватного мессенджера?
- А) Обещание приватности ценно лишь настолько, насколько ему верят; один тайный бэкдор или выдача данных разрушают бренд, поэтому нужны открытый протокол, аудиты, прозрачность
- Б) Доверие ускоряет интернет
- В) Доверие ни на что не влияет
- Г) Без доверия нельзя делать стикеры

**6.** Что должен делать продакт в конфликте приватности и регулирования?
- А) Всегда молча выполнять любые требования
- Б) Всегда игнорировать законы
- В) Обещать максимальную приватность всем и везде, не думая о последствиях
- Г) Честно держать модель угроз: не обещать невыполнимого, согласовывать защиту с позиционированием и законом — вплоть до ухода с рынка, если требования противоречат обещанию

> [!success]- Ответы и разбор
> 1. **А** — доступ к ключам, локализация, бэкдор.
> 2. **Г** — существующая лазейка доступна не только «хорошим».
> 3. **Б** — при настоящем E2EE ключей у компании нет.
> 4. **В** — приватность как ценность бренда + открытость и аудиты.
> 5. **А** — обещание живёт на доверии; бэкдор/выдача убивают бренд.
> 6. **Г** — честная модель угроз, согласование с позиционированием и законом.

## Раздел 21 · Голосовые и видеозвонки: как устроен поток
↩ [[Раздел 21. Голосовые и видеозвонки как устроен поток]]

**1.** Чем звонок принципиально отличается от обмена сообщениями?
- А) Ничем
- Б) Звонок — непрерывный поток в реальном времени, где важнее скорость: лучше потерять кусочек звука, чем ждать; у сообщений наоборот — важнее ничего не потерять
- В) Звонок не требует интернета
- Г) Сообщения всегда важнее

**2.** Что такое «сигнализация» и «медиапоток»?
- А) Сигнализация — установка звонка (кто звонит, приём/сброс), идёт через серверы мессенджера; медиапоток — сам голос/видео, в идеале напрямую между устройствами
- Б) Это одно и то же
- В) Сигнализация — видео, медиапоток — звук
- Г) И то и другое всегда через сервер

**3.** Зачем нужны STUN и TURN?
- А) Шифровать сообщения
- Б) Хранить историю звонков
- В) Замедлять звонок
- Г) Устройства за NAT не видят друг друга напрямую; STUN помогает узнать публичный адрес для прямого соединения, TURN — ретранслятор, когда напрямую не вышло

**4.** Почему для голоса используют UDP, а не TCP?
- А) UDP надёжнее и пересылает потерянное
- Б) TCP не работает с интернетом
- В) UDP не ждёт и не пересылает потерянное — для живого разговора лучше потерять кусочек, чем заморозить звонок ради восстановления (а TCP ждёт и пересылает)
- Г) UDP шифрует, а TCP нет

**5.** Что такое WebRTC?
- А) Название мессенджера
- Б) Стандартная технология для связи в реальном времени: захват звука/видео, кодеки, соединение через STUN/TURN, шифрование — на ней строят звонки в большинстве мессенджеров
- В) Алгоритм шифрования сообщений
- Г) База данных

**6.** Что делают при плохой сети, чтобы звонок не «замёрз»?
- А) Снижают качество на лету, сглаживают джиттер маленьким буфером, мелкие потери маскируют — лишь бы разговор не прерывался
- Б) Полностью прекращают звонок
- В) Пересылают каждый потерянный пакет, пока не дойдёт
- Г) Ничего, ждут восстановления сети

> [!success]- Ответы и разбор
> 1. **Б** — звонок реального времени: скорость важнее надёжности, у сообщений наоборот.
> 2. **А** — сигнализация через серверы, медиа в идеале напрямую (P2P).
> 3. **Г** — из-за NAT: STUN для прямого соединения, TURN — ретранслятор-запаска.
> 4. **В** — UDP не ждёт; лучше потерять кусочек, чем замёрзнуть.
> 5. **Б** — WebRTC: захват, кодеки, STUN/TURN, шифрование.
> 6. **А** — снижение качества, сглаживание джиттера, маскировка потерь.

## Раздел 22 · Групповые звонки и шифрование
↩ [[Раздел 22. Групповые звонки и шифрование]]

**1.** Почему групповой звонок нельзя сделать просто «каждый со всеми»?
- А) Это запрещено провайдером
- Б) Так обязательно теряется звук
- В) В полносвязной схеме каждое устройство отдаёт свой поток отдельно каждому; с ростом числа людей отдача и нагрузка на процессор взрываются — работает только для крошечных групп
- Г) Нужен только один сервер

**2.** Что делает SFU (сервер выборочной пересылки потоков)?
- А) Каждый участник шлёт свой поток ОДИН раз на SFU, сервер пересылает его остальным; устройство отдаёт один поток независимо от размера группы
- Б) Смешивает все потоки в один и рассылает
- В) Шифрует текстовые сообщения
- Г) Хранит запись звонка

**3.** Что делает MCU (сервер-микшер) и в чём минус?
- А) Только пересылает потоки без обработки
- Б) Принимает все потоки, декодирует, смешивает в ОДИН поток и шлёт каждому только его; устройство скачивает один поток (хорошо для слабых), но сервер делает тяжёлую работу — дорого и добавляет задержку
- В) Ничего не делает
- Г) Соединяет всех напрямую

**4.** Когда выбирают MCU, а когда SFU?
- А) MCU всегда лучше
- Б) SFU только для двух человек
- В) Разницы нет
- Г) SFU — стандарт для группового видео (отдал один поток, сервер раздаёт, гибко); MCU — когда у устройств слабый канал или процессор и важно скачивать только один поток

**5.** Почему сервер-микшер (MCU) несовместим со сквозным шифрованием?
- А) MCU работает быстрее
- Б) MCU не нужен интернет
- В) Чтобы смешать потоки, MCU обязан их декодировать — то есть увидеть в открытом виде; это нарушение сквозного шифрования
- Г) MCU прекрасно совместим с E2EE

**6.** Может ли SFU поддерживать сквозное шифрование?
- А) Да, но сложнее: SFU только пересылает пакеты, поэтому медиа можно зашифровать дополнительным слоем под ключ участников — сервер раздаёт, не видя содержимого; смена состава требует перевыдачи общего ключа
- Б) Нет, никогда
- В) Да, безо всяких усложнений
- Г) Только если отключить звук

> [!success]- Ответы и разбор
> 1. **В** — в полносвязной схеме отдача и кодирование растут с группой.
> 2. **А** — SFU: один поток от каждого, сервер раздаёт; устройство отдаёт один.
> 3. **Б** — MCU декодирует и смешивает в один поток; устройству легко, серверу тяжело.
> 4. **Г** — SFU стандарт; MCU — при слабых устройствах/каналах.
> 5. **В** — микширование требует декодирования, это ломает E2EE.
> 6. **А** — SFU + доп. слой шифрования возможен; смена состава требует перевыдачи ключа.

## Раздел 2A · API-ручки
↩ [[Раздел 2A. API-ручки (эндпоинты)]]

**1.** Что такое API-ручка?
- А) Имя сервера в дата-центре
- Б) Адрес плюс одно конкретное действие, по которому клиент дёргает сервер
- В) Таблица в базе данных
- Г) Алгоритм шифрования

**2.** Как понять, какие ручки нужны системе?
- А) Скопировать у конкурента
- Б) Придумать наугад побольше
- В) Пройтись по функциональным требованиям и для каждого описать, как клиент это запросит
- Г) Спросить у базы данных

**3.** Через что идёт доставка сообщений в реальном времени?
- А) Через GET-ручку, которую клиент постоянно опрашивает
- Б) Через DELETE-ручку
- В) Через постоянное WebSocket-соединение — сервер сам толкает сообщение, не дожидаясь запроса
- Г) Через электронную почту

**4.** Как правильно загружать медиа на уровне API?
- А) `POST /media` отдаёт pre-signed URL, клиент льёт файл напрямую в объектное хранилище/CDN, а в сообщении едет ссылка
- Б) Отправить файл целиком в теле `POST /messages`
- В) Сохранить файл прямо в базе сообщений
- Г) Передать файл по WebSocket

**5.** Чем защитить `POST /messages` от дублей при повторной отправке?
- А) Ничем, дубли неизбежны
- Б) Идемпотентным ключом `client_msg_id`, по которому сервер отбрасывает повтор
- В) Полностью запретить ретраи
- Г) Удалять старые сообщения

**6.** Как отдавать историю чата в ручке `GET /messages/{chat_id}`?
- А) Возвращать сразу все сообщения чата
- Б) Возвращать случайные сообщения
- В) Страницами по курсору (`?after={seq}&limit=N`) — тем же, что используется для догона после офлайна
- Г) Только самое последнее сообщение

> [!success]- Ответы и разбор
> 1. **Б** — ручка = адрес + одно действие (`POST /messages` — отправить).
> 2. **В** — набор ручек выводят из функциональных требований: одно действие пользователя — одна ручка.
> 3. **В** — realtime-доставка идёт по WebSocket: сервер толкает сообщение сам, ручки — для разовых запросов.
> 4. **А** — pre-signed URL и прямая загрузка в хранилище/CDN; через API-гейтвей мегабайты не гоняют.
> 5. **Б** — идемпотентный `client_msg_id`: at-least-once + дедуп = эффект «ровно один раз».
> 6. **В** — пагинация по курсору, а не «отдать всё»; тот же курсор — основа догона.

## Раздел 8A · Модель данных
↩ [[Раздел 8A. Модель данных (схемы таблиц)]]

**1.** По какому ключу обычно шардируют таблицу `messages`?
- А) По `user_id` отправителя
- Б) По `chat_id` — все сообщения чата на одном шарде, запрос «последние N чата» бьёт в один шард
- В) По времени создания
- Г) Случайно

**2.** Почему членство (`chat_members`) держат отдельно от `messages`?
- А) Так требует SQL
- Б) Чтобы было больше таблиц
- В) Разная нагрузка и паттерн доступа: fan-out читает список участников, история — бесконечный append
- Г) Это одна и та же таблица

**3.** Что задаёт порядок сообщений в чате на уровне данных?
- А) Время на сервере
- Б) Монотонный per-chat `seq`, по нему сортируют; разрыв `seq` = пропуск
- В) Алфавитный порядок
- Г) `id` отправителя

**4.** Как в схеме поддержана прочитанность при нескольких устройствах?
- А) Один флаг «прочитано» на сообщение
- Б) Отдельный курсор `last_read_seq` на каждую пару (чат, устройство)
- В) Никак, прочитанность не хранят
- Г) В таблице `users`

**5.** Как хранится медиа в модели данных?
- А) Блобом прямо в таблице `messages`
- Б) В объектном хранилище/CDN, а в `messages`/`attachments` — только ссылка, тип, размер, превью
- В) В таблице `users`
- Г) Только в оперативной памяти

**6.** SQL или NoSQL — что для истории сообщений, что для аккаунтов?
- А) Всё в одной SQL-базе
- Б) История — NoSQL-стиль (поток записей, рост, шардирование); аккаунты, контакты, членство — SQL (строгие связи)
- В) Всё в NoSQL
- Г) Всё только в кэше

> [!success]- Ответы и разбор
> 1. **Б** — шардируем по `chat_id`: локальность чтения истории чата.
> 2. **В** — членство и история живут по-разному: список участников для fan-out против бесконечного append истории.
> 3. **Б** — порядок по монотонному per-chat `seq`, не по времени; разрыв = обнаруженный пропуск.
> 4. **Б** — курсор `last_read_seq` на каждое устройство, чтобы догоняли независимо; сервер — источник правды.
> 5. **Б** — медиа ссылкой: файл в хранилище/CDN, в таблице — только метаданные и URL.
> 6. **Б** — история в NoSQL-стиле, аккаунты и членство в SQL: правильный инструмент под тип данных.
